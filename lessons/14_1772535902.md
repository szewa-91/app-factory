# Lesson Learned: Multi-Layer Verification in Autonomous Systems

## Technical Insights
1. **Internal vs. External Health Checks**: Relying solely on external `curl` checks (via Traefik/Domain) can be brittle due to DNS or networking delays. Implementing internal checks (`docker compose exec ... curl http://localhost:3000`) provides a more immediate and accurate assessment of the application's readiness.
2. **Container State vs. Status**: When parsing `docker compose ps --format json`, the `State` field (e.g., `running`) is more reliable for programmatic checks than the `Status` field, which often contains human-readable uptime strings (e.g., `Up 3 minutes`).
3. **Template-Driven Reliability**: Injecting verification scripts at the bootstrap phase ensures that every new project follows the same quality standards. Adding `curl` to the slim production image is a small price (a few MBs) for the significantly improved observability and automated verification it enables.
4. **Graceful Degradation**: Designing the scheduler to call `./verify.sh` *if it exists* allows the system to remain compatible with legacy projects while providing enhanced security for newer ones.
