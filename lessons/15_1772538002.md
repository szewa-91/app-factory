# Lesson Learned: Implementing Context-Rich Self-Healing in Schedulers

## Technical Insights
1. **Diagnostic Context Capture**: Relying solely on log snippets for failure recovery is often insufficient in complex environments (Docker, file system side effects). Capturing `ls -laR` and `docker ps` at the exact moment of failure provides the agent with necessary structural context (e.g., missing files, crashed containers) that logs alone might not reveal.
2. **Recursive Listing with Limits**: Using `ls -laR | head -n 100` provides a quick overview of the directory structure without risking huge output if the project contains many files (like `node_modules`).
3. **Execution-Time Safety**: In autonomous systems with `git reset --hard` safety nets, failure context MUST be captured BEFORE the reset occurs, otherwise, the diagnostic evidence (faulty files) is wiped out before the agent can see it.
4. **Context Window Management**: Providing ~8KB of failure context (`head -c 8000`) is a sweet spot that allows for significant diagnostic data without overwhelming the LLM's context window or tokens.
