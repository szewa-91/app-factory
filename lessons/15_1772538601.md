# Lesson Learned: Implementing Context-Rich Self-Healing in Schedulers (Part 2)

## Technical Insights
1. **Directory Path Robustness**: Always calculate `PROJECT_PATH` explicitly based on the project name. Relying on implicit paths or empty strings can lead to incorrect `cd` behavior and missing files.
2. **Beyond Logs: System Context**: Self-healing is significantly more effective when it understands the *environment* not just the *application*. Including `df -h` and `free -m` helps diagnose common VPS failures like full disks or OOM crashes.
3. **Container Context**: For Docker-based apps, `docker ps -a` and `docker compose logs` are critical. Seeing that a container *crashed* (Exited with code 1) is more diagnostic than just seeing it's not in the running list.
4. **Selective Directory Listing**: Using `find . -maxdepth 2 -not -path '*/.*'` is more effective than `ls -laR` in projects with many files (like `node_modules`). It provides a clear structural overview without cluttering the context window.
5. **Prompt Injection for Self-Healing**: Feeding diagnostic context directly into the agent's prompt with a clear directive ("analyze the logs, directory state, and Docker status above") significantly improves recovery rates on retries.
