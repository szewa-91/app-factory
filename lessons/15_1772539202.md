# Lesson Learned: Implementing Context-Rich Self-Healing in Schedulers (Part 3)

## Technical Insights

1.  **Git Diff as Diagnostic Tool**: Capturing `git diff` before resetting a failed task is invaluable for self-healing. It allows the next agent attempt to see exactly what was changed (and presumably what broke), rather than just the final state or logs.
2.  **Internal vs. External Verification**: Relying only on external HTTP checks is insufficient for complex applications. Integrating an internal `./verify.sh` allows for deeper health checks (e.g., container internal status, database connectivity, internal API response) before exposing the app to external probes.
3.  **Filtered Directory Listing**: While `find` is good for depth-limited searches, `ls -laR` (with appropriate ignores like `.git`, `node_modules`, and `.next`) provides a more standard and informative view of the project's structure and permissions, which helps agents understand the workspace better.
4.  **Database-Driven Configuration**: Schedulers should always fetch project-specific configuration (like `DOMAIN`) from a central database to ensure that verification logic is dynamically adjusted to the project's needs.

## Architectural Improvements

-   **Enhanced Failure Observation**: The `capture_failure_context` function now creates a `PREVIOUS_FAILURE.log` that includes:
    -   Reason and Timestamp
    -   System Resources (`df`, `free`)
    -   Directory Listing (Depth 3)
    -   Git Status and Uncommitted Diff
    -   Docker Status (All) and Compose Logs (if applicable)
    -   Last 100 lines of the task-specific log.
-   **Robust Validation Pipeline**: The validation now follows a 3-step process:
    1.  Internal `verify.sh` (if exists).
    2.  External HTTP Probe (if `DOMAIN` is set).
    3.  E2E Playwright tests (if `tests/e2e` exists).
