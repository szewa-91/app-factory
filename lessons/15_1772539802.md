# Lesson Learned: Implementing Context-Rich Self-Healing in Schedulers (Part 4)

## Technical Insights

1.  **Cleaner Directory Views with `tree`**: Standard `ls -laR` output can be overwhelming for LLM agents. Using `tree -L 3 -I '.git|node_modules|.next'` provides a much cleaner, more visual overview of the project structure, helping agents quickly locate files and understand architecture.
2.  **Container Resource Analysis**: `docker ps` alone only shows if a container is running. `docker stats --no-stream` is critical for diagnosing OOM (Out Of Memory) or CPU-bound failures, providing real-time resource usage snapshots.
3.  **Comprehensive Git Context**: Staged changes (`git diff --cached`) are just as important as unstaged ones. Capturing both, along with the recent `git log --oneline`, allows the self-healing agent to see not just the current diff, but the recent trajectory of changes that led to the failure.
4.  **Multi-Role Log Tailing**: In a developer-auditor workflow, the failure might be in either the implementation or the audit. Capturing both `agent_task_${ID}.log` and `agent_audit_${ID}.log` ensures the next attempt has the full conversation history between roles.
5.  **Infinite Loop Protection**: While self-healing is powerful, it must be bounded. Implementing a `MAX_RETRIES` check (and updating the DB status to `FAILED` instead of just back to `READY`) is essential for production stability to prevent agents from spinning indefinitely on fundamentally flawed tasks.

## Architectural Improvements

-   **Enhanced Failure Observation**: `capture_failure_context` now generates a comprehensive diagnostic report.
-   **Expanded Self-Healing Context**: Increased failure context passed to the next agent from 8KB to 12KB.
-   **Finite State Management**: Tasks now transition to `FAILED` after 3 unsuccessful retries, with detailed reasons recorded in `audit_notes`.
