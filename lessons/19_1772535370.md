# Lesson Learned: Robust Resource Monitoring in Bash

## Technical Insights
1. **CPU Sampling Accuracy**: The first sample of `top -bn1` is often an average since boot or otherwise inaccurate for current load. Using `top -bn2 -d 0.2` (two samples with a small delay) and selecting the second sample (`tail -n 1`) provides a much more accurate reading of the *current* CPU usage.
2. **Robust Extraction**: Extracting specific values from command output (like `id` percentage from `top`) using `sed` or `awk` with fixed positions can be brittle across different OS versions or `top` configurations. Using regex-based extraction (`grep -o "[0-9.]*[ %]*id" | grep -o "[0-9.]*"`) is more resilient to variations in spacing and label formatting.
3. **Graceful Scheduler Exits**: Implementing resource checks *before* task selection in the scheduler allows for graceful skipping of execution without marking tasks as failed, effectively creating a self-throttling mechanism for the VPS.

## Implementation Pattern
- Sample twice for current state (especially for CPU).
- Use `bc -l` for floating point comparisons in shell scripts.
- Log monitored metrics for debugging and visibility.
