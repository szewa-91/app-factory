# Lesson Learned: Implementing Plan-and-Wait Workflows in Bash Schedulers

## Technical Insights
1. **Workflow State Separation**: In a simple task queue, a "Plan-and-Wait" workflow requires a clear distinction between a task being "ready for planning" (READY) and "authorized for execution" (APPROVED). Using an intermediate PENDING_APPROVAL status allows the scheduler to "plan" a task (select it and announce intent) without executing it, providing a natural hook for human intervention.
2. **Robust JSON Parsing in Bash**: When parsing `docker compose ps --format json`, it's more reliable to check the `State` field (e.g., `running`) rather than the `Status` field (e.g., `Up 2 minutes`), as the latter contains dynamic strings that are difficult to match exactly.
3. **SQL SET Syntax Caution**: Automated text replacement (like `sed`) on scripts containing SQL must distinguish between `WHERE status = 'READY'` (which can often be changed to `IN (...)`) and `SET status = 'READY'` (which must remain an equality). Incorrectly replacing `SET` status with `IN` results in invalid SQL.
4. **Port Configuration Consistency**: Deployment verification scripts must be synchronized with the application's actual listening port (e.g., 3001 vs default 3000) defined in `docker-compose.yml` or environment variables to avoid false negatives during health checks.
