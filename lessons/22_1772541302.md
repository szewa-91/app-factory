# Lesson Learned: Optimizing E2E Integration in Autonomous Systems

## Technical Insights

1. **Host vs. Container E2E Testing**: Running Playwright on the host while the application runs in a container is more efficient for ephemeral environments, but requires careful coordination. Using a `PLAYWRIGHT_BASE_URL` override allows testing the *deployed* container via its public Traefik-managed URL, providing a higher-fidelity "smoke test" than internal `localhost` checks.

2. **Wait-for-Public-URL Loop**: Newly bootstrapped applications (especially with Traefik/Docker Compose) can have a delay between the container being "up" and the public URL being reachable/responsive. Adding a retry loop in the verification script to wait for the public URL *before* running Playwright significantly reduces flakiness.

3. **Consolidating Verification Logic**: Avoid redundant test execution by centralizing application-specific verification (including E2E) within a single project-level script (`verify.sh`). The parent scheduler should only call this script and check infrastructure health, preventing overhead and conflicting test results.

4. **SSL and Pending Certificates**: Setting `ignoreHTTPSErrors: true` in the default Playwright configuration is essential for autonomous factories where applications might be tested before Let's Encrypt certificates are fully issued or during development on local domains.
