# Lesson Learned: Source-Level Hardening Beats Post-Failure Recovery

## Technical Insights
1. **Risk-gate fields must be selected and parsed together**: Approval logic that references `priority` without including it in the SQL projection silently degrades safety controls; scheduler selection and gating conditions must be schema-aligned.
2. **Resource guards need tool-absence resilience**: Bash schedulers running across heterogeneous VPS images should treat `top`/`free` parsing as best-effort with safe defaults, otherwise host variance turns protective checks into failure points.
3. **Bootstrap should be failure-atomic across filesystem and DB**: Rolling back both app directories and inserted `projects/tasks` rows on any bootstrap failure prevents dead-end retries and keeps the task queue trustworthy.
4. **Deterministic decomposition fallback preserves policy compliance**: When AI decomposition fails, generating baseline `PRD`/`SPECS` artifacts and seeded atomic tasks keeps the pipeline operational without manual intervention.
5. **Trustworthy verification requires strict TLS and readiness sequencing**: Removing `curl -k`, waiting for runtime readiness before migrations, and single-sourcing `APP_PORT` across templates removes a class of flaky deploy/verify regressions.
