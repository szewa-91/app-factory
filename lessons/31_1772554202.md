# Lesson Learned: Runtime-Proven Hardening Beats Syntax-Only Confidence

## Technical Insights
1. **Risk gates must be projection-complete and type-guarded**: Scheduler approval logic is only trustworthy when SQL selection and Bash evaluation stay coupled (`priority` + `retry_count` selected together with numeric validation before comparisons).
2. **Smoke mode should fail early on infrastructure prerequisites**: Checking `docker`, `docker compose`, and `coolify` network before scaffolding prevents long bootstrap runs from failing late during deploy/verify.
3. **Hardening needs persistent runtime evidence**: Writing smoke execution output to `bootstrap_smoke_<app>.log` creates auditable proof that template changes survive real `deploy.sh` + `verify.sh` execution.
4. **Build context hygiene is a first-order reliability control**: Seeding `.dockerignore` with test artifacts, agent metadata files, and env/log outputs prevents avoidable Docker context bloat and flaky build behavior.
5. **Strict HTTPS probes should constrain protocol and TLS floor**: Using `curl --proto '=https' --tlsv1.2` in verification prevents accidental downgrade paths while retaining certificate validation guarantees.
