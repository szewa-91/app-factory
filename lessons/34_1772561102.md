# Lesson Learned: Canonical Root Docs Must Encode Security-Critical Runtime Contracts

## Technical Insights
1. Root `PRD.md`/`SPECS.md` are not only planning artifacts; they must capture concrete runtime behavior (auth token integrity checks, explicit failure modes, and deterministic API error mappings) or scheduler prompts drift into unsafe implementations.
2. Security hardening in code must be paired with operational hardening in deployment scripts, otherwise fail-closed auth changes can break runtime verification when required secrets are missing.
3. API specs should define both success and not-found/error paths (`P2025` -> `404`) to keep dashboard behavior deterministic for agents and auditors.
