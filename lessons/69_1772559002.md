# Lesson Learned: Parity-First Scheduler Rewrites Need Behavior Contracts, Not Just Module Stubs

## Technical Insights
1. **State-machine parity must be specified as transition rules**: For task schedulers, documenting exact `status` transitions (`READY -> PENDING_APPROVAL`, `IN_PROGRESS -> AUDITING`, retry fallback paths) is more reliable than high-level architecture prose.
2. **Risk-gate correctness depends on schema-aligned typing**: `priority`, `retry_count`, and `status` must be selected, validated, and evaluated together; separating projection from gate logic introduces silent safety regressions.
3. **Lock/PID semantics are core behavior, not implementation detail**: Single-instance lock files and per-project `agent.pid` reconciliation govern correctness under crashes and must be part of the explicit contract.
4. **Resource guards need deterministic fallback order**: CPU/RAM probes should encode parser fallback precedence (`top`/`free` then `/proc`) to keep overload protection stable across heterogeneous Linux hosts.
5. **Wrapper simplicity protects operational compatibility**: Keeping `scripts/scheduler.sh` as a thin `node dist/index.js` launcher preserves cron/systemd integrations while allowing an internal TypeScript rewrite.
