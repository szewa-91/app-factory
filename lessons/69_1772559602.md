# Lesson Learned: Scheduler Specs Need Explicit Hardening Contracts

## Technical Insights
1. **Parity specs must encode branch-level behavior, not summaries**: Lock handling, stale PID reconciliation, approval gating, audit strictness by retry count, and retry/fail transitions should be documented as explicit state transitions to avoid silent regressions in rewrites.
2. **Security parity requires making unsafe legacy patterns illegal**: When migrating from bash, specifications should mandate bound SQL parameters and `spawn/execFile` with `shell: false`; otherwise interpolation risks are accidentally preserved.
3. **Verification confidence comes from a scenario matrix tied to branches**: A fixture-based test matrix that maps each critical branch (`127`, `124`, audit fail, verify fail, race claim, overload skip) to expected DB transitions and log markers is the fastest way to prove behavioral equivalence.
