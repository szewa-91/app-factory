# Lesson Learned: Parity Specs Must Capture Loop Semantics, Not Just Branch Rules

## Technical Insights
1. Scheduler parity requires documenting iteration semantics exactly; in this case, `PENDING_APPROVAL` transitions do not end the cycle and the selector must continue scanning remaining candidates.
2. Schema-first typing is safest when interfaces preserve SQLite column names (`project_name`, `retry_count`, `depends_on`) instead of translating to app-style names, which reduces query and mapping drift.
3. Migration specs are strongest when they lock runtime contracts to concrete artifacts and paths (source shell script line behavior, lock/PID files, `dist/index.js`, wrapper command), making implementation and audits deterministic.
