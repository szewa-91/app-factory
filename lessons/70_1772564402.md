# Lesson Learned: DB Parity Starts With Schema-Literal Typing

## Technical Insights
1. TypeScript DB wrappers for SQLite schedulers stay reliable when interfaces preserve exact column names (`project_name`, `retry_count`, `depends_on`) instead of camelCase remapping at the boundary.
2. Prepared statements with bound parameters across all read/write paths reduce parity bugs and remove accidental SQL interpolation risks during incremental scheduler rewrites.
3. Even scaffold entrypoints should execute a concrete typed path (for example, reading candidate tasks) so compile-time contracts are exercised immediately.
