# Lesson Learned: Scheduler Core-Loop Parity Depends on Explicit Transition APIs

A shell-to-TypeScript scheduler rewrite stays reliable when the DB layer exposes one method per state transition (`CREATED` scan, dependency status reads, `PENDING_APPROVAL` promotion, and atomic claim) instead of generic CRUD helpers. Strict TypeScript settings (`noUncheckedIndexedAccess`) also force safer `/proc` parsers: parse each CPU field with explicit undefined checks rather than tuple-destructuring unchecked arrays. Singleton locks should be created atomically (`wx`) and only remove stale lock files after re-reading unchanged contents to reduce races between concurrent scheduler starts.
