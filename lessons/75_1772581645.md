# Lesson Learned: Bootstrap Recovery When Directory and DB State Are Partially Initialized

## Context
Task 75 was a bootstrap for `personal-website`. When the agent started, the directory already existed with `agent.pid` and `agent_task_75.log` (created by the scheduler), and the project was already registered in `factory.db` with port 3002 (from a previous failed or partial bootstrap attempt).

## Problem
The `bootstrap.sh` script has two hard blockers:
1. It exits if the app directory is non-empty (`ls -A` check)
2. It would fail on `INSERT INTO projects` due to unique constraint violation (project already in DB)
3. `APP_PORT` is hardcoded to 3000 — not configurable via env var

Additionally, `npx create-next-app@latest .` itself refuses to run in a non-empty directory.

## Solution
When the bootstrap script cannot run due to partial prior state, perform a **manual bootstrap** that mirrors the script's steps:
1. Temporarily move scheduler agent files (agent.pid, agent_task_*.log) to /tmp
2. Run `npx create-next-app@latest .` in the now-empty directory
3. Restore agent files after scaffolding
4. Create all config files manually (Dockerfile, docker-compose.yml, deploy.sh, verify.sh) using the correct port from the DB entry
5. Create domain-specific PRD.md and PLAN.md instead of the generic waitlist template
6. Run `git init` and initial commit

## Key Insight
The scheduler always creates `agent.pid` before the agent runs, making a fresh bootstrap impossible without first relocating those files. The bootstrap script should be enhanced to accept `APP_PORT` as an env var override and handle existing agent infrastructure files gracefully. Until then, the manual recovery path is the correct fallback.

## Verification
- `next.config.ts` has `output: 'standalone'`
- `Dockerfile` exposes port 3002
- `docker-compose.yml` routes traffic via Traefik to port 3002
- `deploy.sh` and `verify.sh` pass `bash -n` syntax check
- DB entry: `personal-website|3002|personal.marcinszewczyk.pl`
