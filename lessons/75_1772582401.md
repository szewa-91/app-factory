# Lesson Learned: Audit Failures from Anti-Placeholder Policy — Bootstrap Tasks

## Context
Task 75 bootstrapped `personal-website`. The first attempt passed structure and infra checks but shipped the default `create-next-app` placeholder UI. The auditor (Auditor persona) correctly failed the task on Anti-Placeholder Policy grounds.

## Root Cause
The bootstrap script initializes `create-next-app`, which leaves a placeholder `app/page.tsx` and placeholder metadata in `app/layout.tsx`. The developer agent did not replace these with project-specific content before committing.

## Fix Applied
- `app/page.tsx`: replaced with hero + portfolio preview + footer using real project data (Marcin Szewczyk, 3 portfolio items).
- `app/layout.tsx`: title and description updated; `lang` attribute set to `pl`.
- `playwright.config.ts`: removed `ignoreHTTPSErrors: true` (security/policy violation).
- `.dockerignore`: created to exclude `.env*`, `data/`, logs, and build artifacts from the Docker build context.
- `README.md`: replaced with project-specific content.

## Technical Insights
1. **Anti-Placeholder is a zero-tolerance policy for production code.** Every bootstrap must include a real, functional homepage before the first commit — not after.
2. **Security defaults matter:** `ignoreHTTPSErrors: true` in Playwright silently masks certificate issues in CI. Always remove it unless there is an explicit self-signed cert reason.
3. **`.dockerignore` is as important as `.gitignore`** for security (prevents `.env*` from leaking into image layers) and build determinism.
4. **Retry #1 was fast** because the root cause was clear from `AUDIT_RESULT.md` and `PREVIOUS_FAILURE.log`. The Self-Healing diagnostic context system worked as intended.
