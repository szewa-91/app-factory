# Lesson Learned: Task 75 Retry #2 — Container Not Running Causes verify.sh Failure

## Context
Task 75 bootstrapped `personal-website`. Retries 0 and 1 fixed placeholder content and audit
findings, but verification still failed with "Verify: false, HTTP: true".

## Root Cause
`verify.sh` step 1 uses `docker compose ps --status running --services | grep -qx "app"` to check
container state. The `./deploy.sh` script was never executed in previous retries, so no container
existed. The HTTP: true diagnostic from the scheduler means the public URL returned 200 (possibly
via Coolify or another path), but the local `docker compose` stack had no running container.

## Key Insight
**HTTP: true in the diagnostic context does NOT mean verify.sh will pass.** The scheduler's HTTP
check and verify.sh's container check are independent. An app can be reachable at a public URL
while `docker compose ps` shows nothing — e.g., if Coolify manages the container outside the
local compose stack, or if DNS/CDN serves cached content.

## Fix
Running `./deploy.sh` was all that was needed:
1. Built the Docker image from the local Dockerfile
2. Started the container via `docker compose up -d`
3. Applied pending migrations
4. Container reached `healthy` state on port 3002

## Prevention
- After any code changes in a bootstrap task, ALWAYS run `./deploy.sh` before attempting
  verification. Do not rely on "HTTP: true" as a proxy for container readiness.
- When `verify.sh` reports "Verify: false, HTTP: true", check `docker compose ps` first —
  if no container is listed, the fix is simply to run `./deploy.sh`.
- The deploy step should be treated as part of the bootstrap checklist, not as a post-audit step.
