# Lesson Learned: Task 76 — SPECS.md Creation for personal-website

## Key Insight: URL Structure Must Mirror Original Site

The most critical decision in this SPECS was the blog article route: `/<slug>/` at root level (not `/blog/<slug>/`). This matches the original marcinszewczyk.pl structure and is a SEO/link-preservation requirement. In Next.js App Router, this means creating `app/[slug]/page.tsx` — but this dynamic segment will catch ALL unmatched root-level paths, so it must include `generateStaticParams()` and return 404 for unknown slugs via `notFound()`.

## Static Generation Pattern

With `output: "standalone"` and filesystem-based blog content, the correct pattern is:
- `lib/blog.ts` reads from `content/blog/*.md` using Node.js `fs` module at build time
- `gray-matter` parses frontmatter; `remark` + `remark-html` renders body to HTML string
- `generateStaticParams()` in `app/[slug]/page.tsx` returns all slugs — this enables full static export of article pages

## Dependency Selection

Chose `remark` + `remark-html` over `next-mdx-remote` because: simpler API, no client-side hydration overhead, content is pure static HTML string. `next-mdx-remote` adds value only when MDX (JSX in markdown) is needed.

## Ordering Matters

Implementation order: data/lib layer first → components → pages → build verification. Doing it backwards (pages before lib) wastes time on import errors. Always build the dependency graph bottom-up.
