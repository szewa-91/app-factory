# Lesson Learned: Task 76 Retry #1 — Spec-Level Audit Failures

## Context
Task 76 created `SPECS.md` for personal-website. The first attempt passed structural correctness but failed the Auditor on 4 spec-quality issues.

## Key Insights

### 1. Specs must require sanitization explicitly
When specifying `dangerouslySetInnerHTML`, the spec MUST also mandate a sanitization step (`rehype-sanitize` or equivalent). An Auditor will reject specs that create a clear XSS vector even if content is currently trusted — defense in depth is a spec-level requirement, not just implementation detail.

### 2. Error contracts belong in specs, not just code
Utility functions like `getAllPosts`/`getPostBySlug` need an explicit error-handling strategy in the spec: what happens for malformed input, unreadable files, and edge cases (duplicate slugs, invalid dates). Leaving this to "implementation judgement" is a spec defect.

### 3. No `/* ... */` in spec examples
Placeholder comments in specification code snippets violate the Anti-Placeholder Policy at the spec level. All exported data structures in a spec must have at least one concrete, non-placeholder example — even if minimal.

### 4. Rendering mode must be unambiguous
"Static-first" in the overview combined with "dynamic" in the page table creates architectural confusion. The spec must be internally consistent: pick one rendering mode (SSG/ISR/SSR) per route and state it clearly in a table, with a documented revalidation policy.

## Pattern to Follow
When writing SPECS.md for content-driven sites:
- Security (sanitization) → mandatory in spec, not implementation detail
- Error handling → define strategy (skip+warn vs fail build) per utility
- Data examples → concrete, non-placeholder
- Rendering mode → one consistent table with revalidation policy
