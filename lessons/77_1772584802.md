# Lesson Learned: Task 77 Retry #1 — .dockerignore Scope Too Broad

## Context
Task 77 retry #1 fixed a HIGH audit finding: nav links to /portfolio, /cv, /blog led to 404s. The fix required creating these pages with data from data/*.ts files — but the Docker build failed because .dockerignore contained `data/` (entire directory), which excluded the TypeScript source files needed at build time.

## Key Insights

### 1. .dockerignore must distinguish between runtime data and build-time source
- `data/*.db` (SQLite database) should be excluded — it's mounted from host at runtime
- `data/*.ts` (TypeScript modules) must be included — they are compiled into the app at build time
- Rule of thumb: exclude by file type, not by directory, when a directory contains both source and runtime files

### 2. Navigation routes must exist before deployment
- Any link in a navigation component must have a corresponding page route
- "Coming soon" is acceptable only if the page itself exists (no 404); blank/empty states are real UX, 404s are not
- This should be enforced during the same task that implements navigation, not deferred

### 3. Local build vs Docker build divergence
- `npm run build` locally passed because `data/` was present on the host
- Docker build failed because .dockerignore stripped `data/` from the build context
- Always verify that .dockerignore is consistent with what the build step requires

## Files changed
- `.dockerignore`: `data/` → `data/*.db`
- Added: `data/projects.ts`, `data/cv.ts`
- Added: `app/portfolio/page.tsx`, `app/cv/page.tsx`, `app/blog/page.tsx`
