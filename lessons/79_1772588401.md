# Lesson Learned: Task 79 Retry #1 — CV Section: Tests & Print CSS Scoping

## Context
Retry #1 addressed two audit findings from the initial CV implementation:
1. Missing tests for the `/cv` page and `PrintButton` component
2. Global `@media print` CSS affecting `a` elements app-wide

## Technical Insights

### Testing Next.js server components with Vitest + JSDOM
Server components (no `'use client'`) can be tested directly with `@testing-library/react` in a Vitest/JSDOM environment. Client component imports inside server components (`PrintButton`) are resolved normally in tests — the `'use client'` directive is ignored by the test bundler. Pattern:
```tsx
import CvPage from '@/app/cv/page'
render(<CvPage />)
screen.getByText('Braver IT') // works fine
```

### Mocking window.print in Vitest
```ts
const printMock = vi.fn()
window.print = printMock
fireEvent.click(button)
expect(printMock).toHaveBeenCalledOnce()
```
This pattern works cleanly — no `Object.defineProperty` needed for `window.print` in jsdom.

### Scoping print CSS without CSS Modules
When using Tailwind v4 global CSS and needing to scope print rules, adding a semantic class to the page root (`.cv-page`) and scoping the override to `.cv-page a` is the minimal-change approach. Global rules for `body`, `header`, and `footer` are acceptable across pages; link-specific overrides deserve scoping.
