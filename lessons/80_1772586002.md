# Lesson Learned: Task 80 Retry #1 — Blog Wiring & Frontmatter Schema Compliance

## Context
Task 80 retry #1 fixed three audit findings for the personal-website blog: frontmatter field name mismatch, unconnected markdown content, and a Java syntax error.

## Key Insights

### 1. Frontmatter field names must exactly match SPECS.md schema
The original files used `excerpt` (matching the task description) but SPECS.md line 74 specified `description`. Always read SPECS.md as the authoritative schema source, not the task description text. When a task says "use `excerpt`" but SPECS says `description`, SPECS wins.

### 2. Creating content without wiring it is an Anti-Placeholder violation
Adding markdown files to `content/blog/` without `lib/blog.ts` + `app/[slug]/page.tsx` means the content is never rendered. The auditor correctly flagged this as [MEDIUM]: content exists on disk but is invisible to the app. Always check whether content files need a rendering path before completing a task.

### 3. Next.js 16 `params` is a Promise in Server Components
In Next.js 16 (App Router), dynamic route params (`params`) must be awaited: `const { slug } = await params`. Using `params.slug` directly causes a TypeScript error and runtime warning. Pattern: `async function Page({ params }: { params: Promise<{ slug: string }> })`.

### 4. rehype-sanitize requires explicit `rehype` package
`rehype-sanitize` alone doesn't expose the `rehype()` processor. You need to `npm install rehype` separately. The remark/rehype ecosystem uses separate packages for each processor.

### 5. Markdown-to-HTML pipeline order matters for security
Correct order: `remark().use(remarkHtml, { sanitize: false }).process(content)` → then `rehype().use(rehypeSanitize).process(rawHtml)`. Setting `sanitize: false` in remark-html allows remark to produce richer HTML, then rehype-sanitize with its default schema removes unsafe elements. Never skip the sanitize step when using `dangerouslySetInnerHTML`.
