# Lesson Learned: Task 81 Retry #1 — rehype-sanitize + rehype-highlight coexistence

## Context
Task 81 retry fixed a HIGH security finding: removing `rehype-sanitize` to enable syntax highlighting left `javascript:` and `data:` protocol injection open via markdown links rendered with `dangerouslySetInnerHTML`.

## Root Cause
`rehype-sanitize` with its `defaultSchema` strips all `class` attributes, which breaks `rehype-highlight` (highlight.js attaches language classes like `hljs language-java` to `<code>` and `<span>` elements). The previous developer removed sanitize entirely instead of configuring it.

## Fix Pattern
Extend `defaultSchema` to allowlist `className` on the specific elements that highlight.js touches:

```ts
import rehypeSanitize, { defaultSchema } from "rehype-sanitize";

const sanitizeSchema = {
  ...defaultSchema,
  attributes: {
    ...defaultSchema.attributes,
    code: [...(defaultSchema.attributes?.code ?? []), "className"],
    pre:  [...(defaultSchema.attributes?.pre  ?? []), "className"],
    span: [...(defaultSchema.attributes?.span ?? []), "className"],
  },
};

// Pipeline order: highlight BEFORE sanitize
remark()
  .use(remarkRehype)
  .use(rehypeHighlight, { detect: true })
  .use(rehypeSanitize, sanitizeSchema)  // sanitize last, after classes are applied
  .use(rehypeStringify)
```

## Key Insight
Order matters: `rehype-highlight` must run **before** `rehype-sanitize` so classes are present when the sanitizer allows them. `rehype-sanitize` must run **before** `rehype-stringify` so dangerous HTML never reaches the string output.

## Security Principle
Never use `dangerouslySetInnerHTML` with unsanitized markdown output. Always sanitize after all rehype transforms, not before.
