# Lesson Learned: Task 82 — Homepage Hero Section & Multi-Section Integration

## Context
Implemented the homepage (`app/page.tsx`) with a full aesthetic redesign: hero section with animation, plus Featured Projects, Latest Articles, Skills, and Contact sections.

## Key Technical Insights

### 1. CSS-only hero animation without client components
Used `background-clip: text` + animated `background-position` via `@keyframes` for a shimmer effect on the gradient hero name — zero JS, zero hydration cost, fully SSR-compatible. The pattern:
```css
.hero-name {
  background: linear-gradient(135deg, #f4f4f5 0%, #34d399 40%, ...);
  background-size: 300% auto;
  -webkit-background-clip: text;
  -webkit-text-fill-color: transparent;
  background-clip: text;
  animation: hero-shimmer 6s linear infinite;
}
```

### 2. Layout already provides Navigation + Footer
Since `layout.tsx` renders `<Navigation />` and `<Footer />` around `<main>`, `page.tsx` must NOT include a second footer. The old homepage had a duplicate footer — removing it keeps the DOM clean.

### 3. `getAllPosts()` is synchronous — no async needed
The blog utility returns sorted posts synchronously (no `async/await` required in page.tsx). Calling `.slice(0, 3)` directly on the return value is idiomatic and avoids unnecessary `Promise` wrapping.

### 4. Dot-grid background via inline CSS (Tailwind v4 limitation)
Tailwind v4 doesn't have a built-in dot-grid utility. Defining the `radial-gradient` pattern via `style` prop is simpler and more maintainable than adding a custom plugin or arbitrary value with escaped characters.

### 5. Reusable SVG arrow as a local component
Extracted `<ArrowRight />` as a local function component to avoid repeating the same SVG block 5× across the page. This is the right level of abstraction — one usage would be inline, 5+ usages warrant a local helper.
