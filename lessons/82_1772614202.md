# Lesson Learned: Task 82 retry #1 — Audit Fixes: blog.ts resilience + expanded home tests

## Context
Retry #1 addressed two audit findings from the initial Task 82 homepage implementation:
1. `matter(raw)` in `getAllPosts()` lacked try/catch — a malformed markdown file would crash the homepage at build/render time.
2. `__tests__/home.test.tsx` only had 1 assertion (`h1` exists), missing coverage for all new homepage sections.

## Fixes Applied

### 1. gray-matter parse guard in getAllPosts()
`getAllPosts()` already guarded against unreadable files (`fs.readFileSync` in try/catch), but the subsequent `matter(raw)` call was unguarded. Added a dedicated try/catch that logs a warning and `continue`s to the next file on parse failure. This keeps the homepage resilient against corrupted or non-standard frontmatter.

Pattern to follow: **every external-data parsing step (file read, JSON.parse, matter()) should have its own try/catch with a graceful skip**, not just the I/O step.

### 2. Expanding homepage tests
Went from 1 to 7 tests covering:
- Name in h1
- Hero subtitle + description
- CTA links (View Portfolio, Read CV)
- Section headings (Featured Projects, Latest Articles, Core Technologies, Get in Touch)
- Contact links (email, GitHub)

Key insight: when a Server Component is synchronous and its data dependencies (`getAllPosts`, `projects`) are accessible in the Node.js test environment (filesystem reads, static imports), no mocking is needed — just render and assert static content.

## Key Takeaway
Audit failures around error-handling gaps are easy to miss because the happy path always works. For any data pipeline that feeds a critical render path (homepage, layout), apply **per-step** try/catch rather than wrapping only the I/O boundaries.
