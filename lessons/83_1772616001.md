# Lesson Learned: Task 83 retry #1 — Fix Audit: Dynamic OG Image + Centralized Site Config

## Context
Audit failed Task 83 (SEO implementation) on two findings:
1. [MAJOR] `/og-image.png` referenced in metadata but not present in `public/` — Anti-Placeholder Policy violation.
2. [MEDIUM] `BASE_URL` literal duplicated across 8+ files — drift risk.

## Technical Insights

1. **Dynamic OG image with `next/og` replaces the need for a static asset**: Creating `app/og/route.tsx` that returns `new ImageResponse(...)` serves a real 1200×630 PNG at `/og` without needing a static file. Next.js marks it as `ƒ (Dynamic)` in the build output. No extra npm packages needed — `next/og` is bundled with Next.js 13+.

2. **Never ship metadata that references non-existent files**: Even if an OG image is "planned", committing code that points to a missing asset violates the Anti-Placeholder Policy. Either ship the asset or ship a working dynamic route — never leave a dangling reference.

3. **Centralize shared constants into `lib/site.ts`**: A single `BASE_URL` and `DEFAULT_OG_IMAGE` constant imported by all pages makes domain/URL changes a one-line edit. Duplication across 8 files is a maintenance hazard the auditor will flag.

4. **Do NOT use `export const runtime = "edge"` in standalone Docker deployments**: Edge runtime requires Vercel infra. In standalone Next.js on Docker/Coolify, use default Node.js runtime for route handlers.

5. **`app/[slug]/page.tsx` requires quoting in git commands**: Shell glob expansion breaks `git add app/[slug]/page.tsx` — always quote it: `git add 'app/[slug]/page.tsx'`.
